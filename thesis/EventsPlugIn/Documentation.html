jWebSocket Documentation Guide Lines

Basically, we distinguish between 3 different kinds of documentation for our various target groups.

  • User Guide
  • Administrator Guide (previously called "Installation Guide", but we prefer "Administrator Guide" since this is more personal)
  • Developer Guide

All three documents are explained the following sections.

This is an agile working draft! It is not yet complete and probably will be updated due to our experience during the documentation phase.
It's a first rough guide line and we strive to provide an example for you soon.


User Guide

The goal of this document is to give the user a short explication about each functionalities of the application and after describes step by step how the user can do these functionalities, including photos for each step where does signals that needs to do the user.


  • Overview
  • Features
  • Main functionalities
  • Main contributions
  • Glossary
  • Solution Model
  • Use requirements
  • Main roles
  • Operation of the system
  • Settings
  • Main Rules


  • Introduction/abstract, give a brief description what the general goals and the features of your solution are.
  • What is the conrete purpose of your solution, what's the idea and the technology (in user words) behind it?
  • Why should the user use this product/solution (reasons), what are the benefits compared to other solutions?
  • What is the problem of the user and how does your solution solve that problem?
  • Give the necessary background information and give comprehensible and practival examples to support assumptions and theoretical explanations.
  • If interesting/relevant for the user: What models are used to achieve the solutions.
  • What is required to use the application? (e.g. start/command-line options, required other software to run, for the user level only, no install, config or administrative instructions, these are part of the Administrator Guide)
  • Is the application structured into function modules? If so give an overview over the modules. What are the goals for each module?
  • Show screen shots and explain all functions of the user interface.
  • Explain all action controls (like e.g. buttons) and describe the function behind them.
  • If a complex workflow is necessary to achieve certain goals describe the steps and give examples (e.g. to create a pdf of all employees goto "data/person", select "employees", order by "name", click the "print" button, select "pdf", type filename, open file in pdf viewer etc.)
  • Explain menu items, if such.
  • Explain interactive settings/configuration options (user level/UI only, no xml config or similar here)
  • Explain potential add, insert, delete, update, select, filter, sort, print, im/export, save, load, reset, restart functions.
  • Explain potential interactions and/or conflicts with other applications/plug-ins etc.
  • Explain potential error message and how to deal with it
  • Explain potential known issues and "don'ts" (e.g. "if you confirm the deletion, no recovery/undo will be possible").
  • Explain special things that need to be considered for your solution (e.g. if you don't get a connection to the server, ask your administrator if the required ports are opened, but how to open ports is written in the Administrator guide).


A user guide is good, when a user who never worked with an application becomes able to use all features the solution without the need to contact the support.
Verification: Let another person study your User Guide, let this person explain what (s)he understood in oral word. You passed the test if this person is able to properly use you solution with your personal support/introduction.


Administrator Guide

The goal of this document is to achieve a good process of installation, describes that necessary requirements and all the process of installation and configuration step by step, including photos for each step where does signals that needs to do the administrator.


  • Overview
  • Download
  • Installation environment
  • How install?
  • Hardware
  • Settings
  • How can function?
  • Support


Pure Software

The administrator guide should give answers to the following questions:

  • Where can the software be downloaded? (e.g. Web URL, what is patrt of the download package, how big is the download, what format, special account required to downlad, update service?)
  • What is required before the installation? (e.g. open ports, install database, set environment variables, web server etc.)
  • How is the software do be installed? (e.g. unpacking, installation-wizard, location, modules, integration into other products etc.)
  • What is required after the installation (e.g. create a new user, set up some initial data, required configurations etc.)
  • What rights may need to be granted to the app? (e.g. root account required, new user accounts)
  • Location and scope of the configuration files? (e.g. which config files are needed and where do they have to be located etc.)
  • Which configuration options exist for each setting? (e.g which options per setting, min/max values, recommendations, defaults etc.)
  • Where are the log files located? (e.g. 
  • How can the application be restarted? (e.g. what to do in case of a crash or malfunction)
  • Any known issues and work-arounds? (e.g. what should not be done, what to take care of?)
  • Who can be contacted in case of issues? (e.g. contact persons, phone, fax, email etc.)
  • In case of printed documentation, where is the latest online version available (e.g. Web URL, FTP server)


Software/Hardware Bundles

In case the software is provided together with hardware (e.g.our NFC or Smart Card Apps, or the Arduino Apps) the following questions should also be answered.

  • How is the hardware to be connected (e.g. cables, network, interface, power supply, connectors, plugs etc.)
  • What drivers are required, where can these be obtained and how are these to be installed and configured?
  • What settings need to be done on the hardware to cooperate with our solutions?
  • What differences between the various operating systems?
  • What special settings or preparations before the hardware can be used?
  • How can the hardware be tested separately?
  • Any test software available to test the hardware stand-alone (i.e. without our software), (e.g. does the vendor provide its own tools to exclude issues in our software in case of malfunctions).


The Administrator Guide is good, when a new person who is not yet involved into the solution is able to download, install, configure and operate your solution without external support.
Verification: Let a third person install your solution in their own environment. If this person is able to operate you solution with out your support or introducton you passed the test.


Developer Guide

The goal of this document is to allow developers add or modify functionalities as well as to improve the application or fix errors. In other words: to a achieve a good process of maintenance. This document describes the structure of the application, packages, classes and methods, and all the functionality of the applications, step by step. It refers to the user and administrator manual and gives a reference to the source code for all provided functionality.

Overview

This is summary/overview what is is about. 

  • Project name, author(s) and contributor(s), date, location (where to find), purpose, feature and goals summary
  • List of modifications, incl. Authors (version history)
  • Technical solution abstract

Infrastructure, Model, Approach

This is a summary/overview how it works.

  • Summary of approach, model, technology and implementation (no details here, just summaries)
  • Infrastructure diagram(s), these diagrams show the modules of the solution (top-down view, higher level part on the top, lower level software on the bottom) and their relation or hierarchy (see. e.g. jwebsocket.org, item "infrastructures")
  • Work-flow diagram or data flow-chart where applicable, these diagrams explain more the processes, the order, timing and direction of interactions and interfaces.
  • Development model, just summary or overview here, details in source code section
  • Used patterns (e.g. publish/subscribe, observer or facade pattern), just summary or overview here, details in source code section
  • Definition of events and treatment of aspects, where applicable
  • Events vs. Tokens (EventPlugIn or TokenPlugIn), where applicable, especially in plug-development.
  • Filters, Plugins, Apps, here it is about model and approach, a summary, neither implementation nor configuration details here
  • Client/Server Structure, Javascript, Java, Mobile clients etc.

Requirement and Prerequisites

This is a summary/overview of the essential requirements and prerequites for the solution.

  • Required resources (e.g. databases, network connections)
    media (e.g. hardware (arduino), drivers (rxtx), demo-data, examples, images, videos or other files not within the solutions itself),
    styles (e.g. designs, templates), html pages, databases, configuration files
  • Maven settings, NetBeans plug-ins oder configurations, spring usage, other framework requirements (e.g. jQuery, Sencha, Bindows etc.)

Modules, Structure

This section describes the organizational structure of your solution in functional modules.

  • Packages and other modularization criteria, e.g. spitting in multipe modules (.jars)
  • Project location, folder locations for source and executable code, package structures and file locations, dependencies
  • How is the separation between library (e.g. database access), common high level logic (e.g. db cache sychronization), business/service logic (e.g. algorithms), presentation logic (e.g. application, demo) realized?
  • If heterogenous environments: Explain which modules are implemented in which languages, IDE's, how they interact (e.g. JavaScript on the Client exchanges JSON packets with Java Token Plug-in on the server side).
  • Explain packages per module and classes per package   
  • How was extensibility in a cluster considered (e.g. shared, centralized/de-centralized components, e.g. database layer vs. logic-layer)?

Source Code

This section desribes the details of the source code of your solution.

  • Packet structure, name of the class that the class is extended
  • SVN path, project root directory, branches, required merges in case of updates/fixes
  • Used standards or common coding and naming conventions, if matching refer to pages, if deviating please give reasons/explanations.
  • Used inline documentation standards, should be JavaDoc standard or JSDoc like in jWebSocket.js
  • How is the inline documentation  done in non-Java(Script) codes (e.g. C#, Python etc.)?
  • Re-Usable code parts (libraries), how can other software access/benefit from the shared parts of your solution(s), what's you API, what API's do you use.
  • The complete list of files is given in the Reference section.
  • Error and Exception handling is part of the Test and Quality Assurance section.
  • Profiling provisions also have their own section.

Interfaces (if applicable)

This section describes the APIs to other modules of the product.

  • Explain the communication with the lower level components (e.g. which feature of libraries are used in the lower software stack)
  • Explain the communication with the higher level components (e.g. which features does the solution provide to the upper software stack)
  • Explain the relation with other components on the same level (e.g. which features of other plug-ins are utilized, e.g. SMS by the WatchDog)
    "Public" interfaces (especially for plug-ins: which functions and be called via our invoke mechanism from other plug-ins or apps)
  • Explain data formats (e.g. JSON, XML etc.), network protocols (e.g. TCP/UDP/HTTP or SMTP/IMAP/POP3), programming techniques (RMI, RPC, methods, callbacks, events etc.), programming models (e.g. (D)COM, Java .jars, JNI etc.), technologies (e.g. )
  • Describe the API in words here, give examples were helpful. Details to classes, methods, arguments can be written in the reference section.
  • Used standard APIs (e.g. WebServices (SOAP), REST or similar)

Frameworks, Libraries and Tools

This section describes which third party tools are utilized for which purpose(s) and how they are integrated.

  • General dependencies (maven and others), required frameworks libaries, our own ones and third party ones if required.
  • dependency tree, do the third party tools have also further dependencies.
  • Links to all used third party tools to documentation, download and update pages (as far as available)
  • Expose the license(s) of all used third party tools! How do licenses affect distribution model of the entire product?
  • What about the maintainability and possibility to test the used tools? Quality and potential effects of the tools to the solution (our product)
  • List all required third party libs and related files in Reference section (required for release and deployment automation).

Database and Persistence (if applicable)

This section describes how data is stored, either in databases or other persistence instances.

  • Database info: type of database, list of tables, their purpose/meaning, schema, install scripts, database user rights, DB schema, data model, access and user rights if needed/applicable
  • Host info: (server/port/protocol/other service options), field names and meanings, data types, relations (graphical model), constraints, indexes, sequences, triggers, stored procedures, backup and restore procedures and locations
  • SQL, NoSQL, XML, ini/properties, text files or other approach? Explain the methods of persisting data and reasons for the selection.
  • Graphical model of tables, relations and constraints

Hardware (if applicable)

This describes the hardware the solution requires or is based on.

  • What hardware is required, (e.g. mobile devices, arduino, sensors, peripheral device etc.), here it is about the technical hardware details of the devices. 
    Hardware configuration or drivers which need to be installed or configured are part of the Administrator Guide!
  • Connection Technology (e.g. connectors/interfaces like USB, serial, firewire and required cables etc.)
  • Hardware on certain operating system related considerations and firmware.
  • APIs of the hardware (references to vendor documentation). 

Security

This section describes all security aspects of the solution. Here it is more about the implementation rather than the configuration, which is part of the Administrator Guide.

  • How are credentials maintained (e.g. to databases or to other servers like Mail-Server or Twitter).
  • What about encyrption/SSL options, how to ensure security for the users.
  • How is the user data privacy ensured?
  • How is this service protected against spam or other attacks?
  • What validation criteria is applied and how?
  • Filters, black- and whitelists, firewall considerations (open ports and protocols)
  • What other mechanism have been applied in security aspects (or are supposed to be applied in the future)?
  • What known security issues do currently exist and are these acceptable for a production release? What protection is possible? Are workarounds available?

Tests, Quality Assurance, Continuos Integration

This section describes the execution of automated and manual tests for quality ensurance. Right now we do not yet have continuos integration, so this is still pending.

  • Test cases, functional (JUnit, Jasmine) and UI tests (iMacros, pending).
    Which automated tests are available, how are they invoked, how are the test cases integrated into our deployment test suite? Where are the reports?
    Are there known issues (traps, also refer to bug-list), what are the reasons, is this approved and what can be done in case of these issues (work-arounds)?
  • What about the error and potential issue handling (e.g. database not available, network connection broken other resources do not exist).
  • What about the general error and exception handling?
  • If required (because not (yet) covered or not coverable), which manual tests can/have to be run before deployment?

Profiling

This section describes the methods and strategies for continous improvements.

  • "Improvements" means provisions for higher speed, less memory consumption, simpler maintainability or similar, which do not affect the functionality in general (e.g. "the solutions uses a quicksort instead of a bubblesort", or "in future the solution should use MongoDB instead of MySQL because..."). It does not mean bug-fixing. 

Reference

This is a structured lookup document, here it is about a complete reference of details, just with keywords or short sentences rather than much text,

  • Complete JavaDocs or JavaScriptDocs (inline documentation) are mandatory for publishing (classes, constants, variables, methods, arguments, result, examples, properties, flags), all classes and methods in all languages should be fully documented for efficient knowledge management and maintainability.
    The JavaDocs and JavaScript Docs as well as other automatically generated documention which is targetted for online-distribution (on our website), does not need to be printed in the Developer Guide, here a reference to the online area is sufficient.
  • A "Token Reference" is mandatory for the Developer Guide (see e.g. the Channel Documentation on our WebSite)
  • List of files, as far as not automatically generated by any inline documentation tool, like JavaDocs. (e.g. a certain set of configuration files and their purpose), for configuration files, list of settings and possible options. This is also important for the packaging and deployment of the solution.


A Developer Guide is good, when a third person who is not involved into the development becomes able to understand and maintain the application.
Verification: Let another developer read your Developer Guide and either implement/add a new feature or fix a bug. If he achieves that in a reasonable time you passed the test.
